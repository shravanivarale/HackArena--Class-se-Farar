1. PROJECT OVERVIEW
================================================================================

VitalScore Finance is an AI-driven, gamified financial wellness platform designed
for India's 500M+ UPI users. It transforms personal finance management from a
boring spreadsheet exercise into an engaging, game-like experience backed by
blockchain-verified trust.

CORE PHILOSOPHY:
  - "Invisible Blockchain" — Users see "Verified ✓" badges, not crypto jargon
  - Behavioral Finance — Uses loss aversion, social proof, and streaks
  - AI-First Categorization — Transactions auto-categorized with confidence scores
  - Real Money Stakes — Optional escrow-based challenges with actual financial commitment

TARGET USERS: Young Indian professionals (18-35) who use UPI daily but lack
financial discipline. The platform turns saving money into a competitive,
rewarding experience.

5.1  AUTHENTICATION (Login / Signup)
────────────────────────────────────────────────────────────────────────────────

USER FLOW:
  1. User opens app → sees Login page (no sidebar/topbar)
  2. Enters email + password → clicks "Sign In"
  3. If account exists: logged in, redirected to /dashboard
  4. If not: clicks "Sign Up" → enters name, email, phone, password
  5. Account created → auto-logged in → welcome notification shown

ARCHITECTURE FLOW:
  Frontend (Login.tsx / Signup.tsx)
    → POST /auth/login or POST /users (via AppContext)
    → Auth Service (port 3002) validates credentials
    → Returns JWT token + user object
    → AppContext stores user in state + localStorage
    → ProtectedRoute wrapper allows access to app pages

HOW IT WORKS:
  - Auth Service uses Helmet for HTTP hardening, rate-limits to 100 req/15min
  - Passwords hashed with bcrypt before storage
  - JWT tokens for session management with refresh capability
  - OAuth2 integration available (Google, etc.)
  - GRACEFUL FALLBACK: If backend is down, stores users in localStorage
    and validates locally (essential for hackathon demo)
  - AppContext.isAuthenticated controls route access via ProtectedRoute component
  - Unauthenticated users are redirected to /login via Navigate component
  - On logout, localStorage is cleared and user is redirected to /login

TECH:
  Frontend: React forms with controlled inputs, useCallback for handlers
  Backend:  Express + Helmet + CORS + JWT + bcrypt + rate-limit
  Storage:  PostgreSQL (user_profiles) + localStorage (fallback)


────────────────────────────────────────────────────────────────────────────────
5.2  VITALSCORE ENGINE (Financial Health Scoring)
────────────────────────────────────────────────────────────────────────────────

USER FLOW:
  1. User's financial score (0-1000) is displayed in TopBar at all times
  2. Score updates in real-time as transactions are added/categorized
  3. Score bands determine UI color theme:
     - Vital Elite (800+):     Green #00d4aa
     - Vital Strong (600-799): Blue  #3b82f6
     - Vital Warning (400-599):Yellow #ffc107
     - Vital Critical (200-399):Orange #ff6b35
     - Vital Emergency (0-199): Red  #ff4757

ARCHITECTURE FLOW:
  Transaction Added/Categorized
    → AppContext.calculateScore() triggered
    → Analyzes 3 months of transaction history
    → Weights: 60% necessity ratio + 40% savings ratio
    → Adds streak bonus (up to +50) and challenge bonus (+25)
    → Subtracts debt penalty
    → Normalized to 0-1000 range
    → Score state updated → UI re-renders everywhere

  Backend (Score Engine Service, port 3004):
    → Listens to SQS transaction events
    → Micro-updates within 60 seconds of new transaction
    → Nightly batch recalculation at 02:00 (3-month rolling average)
    → Stores snapshots in score_snapshots table + InfluxDB

HOW IT WORKS (Formula):
  1. Gather all transactions from past 3 months
  2. Classify into: Essential (E), Discretionary (D), Income (I)
  3. necessity_ratio = E / (E + D)  — How much of spending is on essentials
  4. savings_ratio = (I - E - D) / I  — How much income is saved
  5. raw_score = (0.60 × necessity_ratio + 0.40 × savings_ratio) × 1000
  6. final_score = raw_score - debt_penalty + streak_bonus + challenge_bonus
  7. Clamped to [0, 1000]
  8. Band assigned based on score range

SPECIAL FEATURES:
  - Emergency Mode: If essential spending > 90%, weights shift to 80% essential
  - Anomaly Detection: Flags spending > 300% of 90-day average
  - Inflation Adjustment: CPI data applied for fair comparison
  - Score trajectory: IMPROVING/DECLINING/STABLE tracked over time

TECH:
  Frontend: calculateScore() in AppContext, real-time useMemo in Dashboard
  Backend:  Score Engine (port 3004) + SQS events + node-cron nightly batch
  Storage:  PostgreSQL (score_snapshots) + InfluxDB (score_events)

5.4  SMART NUDGE (AI Transaction Categorization)
────────────────────────────────────────────────────────────────────────────────

USER FLOW:
  1. When a transaction is added (manually or via bank sync), it may arrive
     without a category (needsLabel = true)
  2. Dashboard shows a yellow banner: "Smart Nudge — Categorize Your Spending"
  3. Each nudge shows:
     - "Did you just spend ₹450 on Zomato?"
     - Suggested category: "DiningOut" (87% confidence)
     - Time ago: "2 hours ago"
  4. User can:
     - Click "Confirm" → accepts AI suggestion → earns +12 XP
     - Click "Change ▾" → dropdown of 16 categories → picks correct one → +12 XP
  5. Transaction is now categorized, score recalculates

ARCHITECTURE FLOW:
  New Transaction (needsLabel=true)
    → Dashboard.tsx useMemo generates nudge objects
    → For each: analyzes amount + merchant name patterns
    → Assigns category with confidence score
    → User action triggers labelTransaction()
    → AppContext updates transaction, adds +12 XP to user
    → Score recalculated with properly categorized data

  Backend (AI Categorization Service, port 3007):
    → Two-layer classification:
       Layer 1: Rule-based (merchant name matching) → 95% of transactions
       Layer 2: Groq LLM (BERT-based) for low-confidence (<70%) cases
    → Returns: { category, confidence, alternatives[] }
    → Rate limited to 200 req/15min

HOW IT WORKS (Frontend ML-like Logic):
  Amount-based heuristics:
    < ₹100  → Dining Out
    < ₹500  → Dining Out (or Transportation)
    < ₹2000 → Shopping (or Bills)
    < ₹5000 → Bills (or Shopping)
    ≥ ₹5000 → Housing (or Bills)

  Merchant name pattern matching:
    zomato, swiggy, food     → Discretionary.DiningOut
    uber, ola, metro         → Essential.Transportation
    amazon, flipkart, myntra → Discretionary.Shopping
    netflix, spotify, hotstar→ Discretionary.Subscriptions
    dmart, bigbasket         → Essential.Groceries

  Confidence = 0.55 + random(0.35)  (simulates ML confidence range)

CATEGORY TAXONOMY (16 categories):
  Essential:      Groceries, Transportation, Bills, Housing, Healthcare,
                  Education, Insurance, EMI
  Discretionary:  DiningOut, Shopping, Entertainment, Subscriptions,
                  Travel, PersonalCare
  Savings:        Investment, EmergencyFund

TECH:
  Frontend: useMemo in Dashboard.tsx, pattern matching, dropdown selector
  Backend:  Groq API (LLM), BERT model, rule-based classifier
  Reward:   +12 XP per categorization

5.5  SUBVAMPIRE (Ghost Subscription Detection)
────────────────────────────────────────────────────────────────────────────────

USER FLOW:
  1. System automatically scans transaction history for recurring charges
  2. If a subscription hasn't been used in 25+ days, it's flagged as "ghost"
  3. Dashboard shows purple "SubVampire Alert" banner:
     - "Netflix — ₹199/mo — Unused 45 days — 80% ghost"
     - "Spotify — ₹119/mo — Unused 32 days — 60% ghost"
  4. For each ghost, user can:
     - "Cancel & Save ₹2,388/yr" → marks as cancelled, creates CREDIT transaction,
       logs blockchain proof, earns XP
     - "I use this" → marks as verified, removed from ghost list
     - Clock icon → snoozes for 7 days
  5. Total annual waste shown: "Draining ₹2,500/mo (₹30,000/yr)"

ARCHITECTURE FLOW:
  Transaction History
    → detectGhosts() function runs on transaction changes
    → Groups transactions by merchant name
    → Filters for consistent amounts (within 15% variance → subscription pattern)
    → Checks last usage date (>25 days = potential ghost)
    → Assigns ghostScore: 30 base + days-based bonus
    → Results stored in AppContext.ghosts state

  When user cancels:
    → Ghost status → CANCELLED
    → CREDIT transaction created (the saved amount)
    → Blockchain proof logged (type: ghost_kill)
    → Notification: "Ghost Killed! Cancelled X — saving ₹Y/year!"
    → Score recalculated (more savings = higher score)

HOW IT WORKS (Detection Algorithm):
  1. Group DEBIT transactions by merchant name (lowercase)
  2. Filter merchants with 2+ transactions
  3. Check amount consistency: all amounts within 15% of average
  4. Calculate days since last transaction
  5. If >25 days idle:
     ghostScore = 30 + (>60 days: 50, >30 days: 30, else: 10)
     Cap at 100
  6. Display sorted by ghostScore (highest = most likely ghost)

  ↓

  ↓

  ↓

  ↓

  ↓
final_score = clamp(raw_score - debt + streak + challenge, 0, 1000)

  - AES-256 encryption at rest for all sensitive data
  - TLS 1.3 for all data in transit
  - Bank connections are READ-ONLY (no write/payment permissions)

API SECURITY:
  - Helmet.js: Disables X-Powered-By, sets secure headers
  - CORS: Configurable origin whitelist
  - Rate Limiting: 100-200 req/15min per service
  - JWT: Token-based auth with refresh rotation
  - Request IDs: Unique ID per request for tracing

BLOCKCHAIN SECURITY:
  - Escrow contracts deploy with explicit safety guardrails
  - "Do NOT proceed if stake cannot be locked" — user sees clear error
  - Non-transferable NFTs prevent score manipulation/trading
  - IPFS stores only metadata hashes (no PII on-chain)
  - Smart contracts verified on Algorand TestNet before MainNet

INPUT VALIDATION:
  - All API endpoints validate required fields
  - Razorpay webhook signatures verified with HMAC
  - Phone numbers validated before Twilio API calls
  - Amount validation: positive numbers, reasonable ranges

AUDIT TRAIL:
  - audit_logs table captures all significant actions
  - Blockchain proofs provide independent verification
  - InfluxDB retains 1 year of time-series data
  - Request logging with structured format

PRIVACY:
  - No financial data stored on blockchain (only hashes)
  - User consent required for: escrow, squads, data sharing, B2B
  - Notification frequency configurable: Essential/Standard/Full
  - Logout clears all local state
